@AGENTS.md

## What this is

Nebulion — an NSE/BSE Indian stock research web app (Next.js App Router). Search a company, then browse Overview, SWOT, P&L, Balance Sheet, Cash Flow, Peers, Ratios, News, and Depth tabs at `/company/[symbol]/*`. Standalone tools live off the "Tools" nav dropdown: `/compare` (pick companies to compare), `/industry` (browse a whole sector), and `/screens` (save/run conditional stock screens).

This `web` app is the frontend/API layer of a larger system. A separate **ai-service** (reached via `AI_SERVICE_URL`, default `http://localhost:8000`) owns background scraping jobs and the Angel One live-tick WebSocket bridge; this repo only calls into it, never runs it.

## Commands

- `npm run dev` — start the dev server (Turbopack). Auto-picks the next free port if 3000 is taken — check the terminal output for the actual URL.
- `npm run build` — runs `prisma generate && next build`. Always run this (or at least `npx tsc --noEmit`) before pushing; the Prisma client is generated code and gitignored, so a plain `next build` without `prisma generate` first will fail on a fresh checkout with `Module not found: Can't resolve '@/lib/generated/prisma/client'`.
- `npm run lint` — ESLint.
- `npx prisma generate` — regenerate the Prisma client after editing `prisma/schema.prisma` (also runs automatically via `postinstall` and `build`).
- `npx prisma migrate dev --name <name>` — create and apply a new migration.
- `npx prisma db seed` — seed the `Company` table from `data/company-master.json` (runs `prisma/seed.ts` via `tsx`, per `prisma.config.ts`). `POST /api/admin/seed-companies` does the same over HTTP (send header `x-admin-secret: <value>` if `ADMIN_SEED_SECRET` is set).
- There is no test suite/framework configured in this repo.

Required env vars (see `.env`, not committed): `DATABASE_URL`, `DIRECT_URL` (session-mode pooler, used for migrations), `AI_SERVICE_URL`. Optional: `ADMIN_SEED_SECRET` to gate the admin seed endpoint.

## Architecture

**Data flow / caching model.** Postgres (via Prisma) is a cache in front of Yahoo Finance (`lib/yfinance.ts`, using `yahoo-finance2`), not the source of truth:
- `app/company/[symbol]/layout.tsx` looks up `Company` by symbol first. If missing, it live-fetches a quote + company info from Yahoo Finance and **upserts** a `Company` row so the sub-tabs have something to attach cached data to; if the live fetch also fails, it renders an inline error instead of the tabs.
- Each statement API route (`app/api/companies/[symbol]/{income-statement,balance-sheet,cash-flow}/route.ts`) follows the same shape: try the DB (`FinancialStatement`) first; on a cache miss, fetch live from Yahoo Finance for an immediate response **and** fire `triggerScrape()` (`lib/ai-service-client.ts`) to ask ai-service to backfill the cache for next time. `triggerScrape` is intentionally fire-and-forget (not awaited, swallows its own errors) — never make it blocking.
- Routes that resolve a Yahoo ticker must look up `company.exchange` from Prisma first and pass it through (`getYahooTicker(symbol, exchange)` defaults suffix-less symbols to NSE otherwise) — most routes do this via `const company = await prisma.company.findUnique(...)` before calling into `lib/yfinance.ts`; don't skip it when adding a new route, or BSE-only symbols silently resolve against the wrong exchange.

**Live ticks.** `/api/companies/[symbol]/live` is a thin SSE proxy to ai-service's Angel One stream (never talks to Angel One directly). The client hook `lib/hooks/use-live-quote.ts` degrades silently to `connected: false` when the stream isn't available — treat that as normal, not an error state, in any UI that consumes it.

**Peer comparison.** `lib/peer-columns.ts` is the single source of truth for every comparable metric (~38 entries: key, label, group, `format`, `invert`/`neutral` flags, plus `rawFraction` for values Yahoo reports as 0.15-for-15%). It drives the column editor, the comparison table, and the dashboard uniformly across the Peers tab, `/compare`, `/industry`, and `/screens` — add a new comparable metric there once, not in N places. `toChartValue()`/`formatPeerValue()` are the canonical raw-to-display-unit converters (₹ Cr, %, ratio) — reuse them instead of re-deriving units. The dashboard (`components/company/peer-dashboard.tsx` / `compare-dashboard.tsx`) picks a bar ranking chart vs. a share-of-total pie/donut per column based on `format` (only absolute `cr`-money metrics are pie-eligible, and only when every company's value is non-negative — see `isShareChartEligible`).

**Yahoo's undocumented equity screener** (`lib/yfinance.ts`, `screenEquities`/`screenBySector`) is what powers `/industry` and `/screens`'s market-wide discovery — it's a reverse-engineered POST to `https://query1.finance.yahoo.com/v1/finance/screener` requiring a crumb/cookie handshake (`getYahooCrumb`), since the `yahoo-finance2` npm client only exposes GET-based predefined screens. Two load-bearing facts: it caps page size at ~250 (`SCREEN_PAGE_SIZE`), and `region=in` lists every company **twice** (once per NSE `.NS` / BSE `.BO` listing) — `resolveScreenedMetrics` does the root-ticker + normalized-name dedup that every caller (`getCompaniesBySector`, `getScreenUniverse`) relies on. Screening real-time against the *entire* market (~4,000 distinct companies) isn't feasible live (Yahoo rate-limits, and it's too slow per-request) — `getScreenUniverse` instead pages up to a bounded target (currently 500 distinct companies) and callers paginate the *matches*, not the universe.

**Stock screener query language** (`lib/screener/`) is a small dependency-free parser/evaluator, not backed by any external query-string library: `tokenizer.ts` → `parser.ts` (recursive descent, positioned AST) → `evaluator.ts` (`compileScreen`/`evaluateRow`). Metric names resolve against `lib/peer-columns.ts` (labels, camelCase keys, and short aliases in `metrics.ts`), and conditions are evaluated in **display units** via `toChartValue()` so `Market Cap > 1000` means "> ₹1,000 Cr" with no unit-inference step. A comparison touching a missing/null metric evaluates to `false` (excludes the company) rather than erroring. `Sector = 'X'` is special-cased: it must be a top-level `AND` conjunct (it selects `getScreenUniverse`'s candidate pool, not a per-row predicate) — `evaluator.ts`'s `extractSector` enforces this. Saved screens persist to `localStorage` (`lib/screener/storage.ts`, key `nebulion:screens`) since there's no user/auth model — see below.

**"Tool" pages share one shape.** `/compare`, `/industry/[sector]`, and `/screens/new` (and the Peers tab) all follow: a server `page.tsx` wrapping a `"use client"` view in `<Suspense>`, SWR (`lib/swr-fetcher.ts`'s `fetcher`/`extractErrorMessage`) for data, and **URL search params as the source of truth** (`router.replace(..., { scroll: false })`) rather than component state — this is what makes "Copy link" reproduce the exact view. When adding a new filter/column/page control to one of these, follow this URL-state pattern rather than `useState`.

**No auth/user model exists anywhere in this app.** Any "save this for later" feature (peer-column selection, saved screens) persists to `localStorage` on the client, guarded with `typeof window === "undefined"` checks and try/catch (see `app/company/[symbol]/peers/page.tsx`'s `loadStoredColumnKeys` or `lib/screener/storage.ts`) — don't reach for a Prisma model or cookie-based session for this kind of state.

**Prisma client generation.** The generated client lives at `lib/generated/prisma` (custom `output` path in `schema.prisma`, not the default `@prisma/client`), and that directory is gitignored. Anything that constructs a `PrismaClient` directly (`lib/db.ts`, `prisma/seed.ts`) must import from `@/lib/generated/prisma/client` and pass the `PrismaPg` driver adapter (`@prisma/adapter-pg` + `DATABASE_URL`) — this schema uses Prisma 7's driver-adapter model, not the legacy query engine binary.

**Company master seeding has three parallel implementations** that all upsert the same `data/company-master.json` into `Company`: `prisma/seed.ts` (Prisma, `prisma db seed`), `scripts/seed-master-list.js` (raw `pg` pool insert, uses `DIRECT_URL`), and `app/api/admin/seed-companies/route.ts` (HTTP, uses the Prisma client). They're not kept in sync automatically — if you change the upsert shape, update all three or consolidate them.

**Client pages under `app/company/[symbol]/*`** unwrap the async `params` Promise via a manual `useEffect` (`const { symbol } = await params` inside an effect) rather than React's `use()`. This is a known deviation from this Next.js version's own documented pattern for Client Components (see `AGENTS.md`) — match the existing style in a file you're editing, but prefer `use()` in new code (e.g. `app/company/[symbol]/peers/page.tsx` already does).

**UI components** are shadcn (`components.json`: style `base-nova`, base-ui primitives — not Radix, note `@base-ui/react/tooltip` etc. in `components/ui/`), Tailwind v4 (`app/globals.css`, tokens in an `@theme` block including `--chart-1`..`--chart-8` for theme-aware categorical color, dark mode via `next-themes` with `defaultTheme="dark"`, `enableSystem={false}`). Charts are Recharts; `lib/colors.ts` (`CHART_COLORS`, `gainLossText`/`gainLossBg`) and `lib/format.ts` (`formatINR`, `formatCr`, `formatPercent`) are the shared formatting/color helpers — reuse them rather than hardcoding currency strings or colors in a new chart/table. Base-ui gotchas hit in practice: `Button` needs `nativeButton={false}` when its `render` prop swaps in a non-`<button>` element (e.g. `<Link>`); `DropdownMenuLabel` throws (`MenuGroupContext is missing`) unless wrapped in `DropdownMenuGroup`; the shared `Input` (`components/ui/input.tsx`, wrapping base-ui's `Field.Control`) can enter an infinite re-render loop under rapid successive `onChange` on a bare controlled text field — prefer a plain native `<input>` (styled to match) for simple one-off text fields rather than debugging that further.
